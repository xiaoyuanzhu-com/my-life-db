# i18n Phase 3 — Error-Code Envelope + Handler Migration + `useErrorMessage`

**Goal:** Stop rendering raw English backend error strings in the UI. Introduce a machine-readable error `code` on user-facing errors, translate error messages via `errors.json`, and provide a frontend helper that reads backend errors in a single consistent way.

**Architecture:** The backend's `ErrorResponse` envelope (`{ "error": { "code", "message", "details" } }`) is **already defined** in `backend/api/response.go` with helpers (`RespondBadRequest`, `RespondNotFound`, etc.) but **zero handlers use it yet**. Phase 3 adopts the envelope on ~10 high-visibility handlers with **semantic error codes** (not the existing HTTP-status-adjacent coarse codes), adds `errors.json` catalogs, and adds an `errorMessage(err, t)` utility + `parseApiError(response)` parser on the frontend.

The migration is gradual — legacy handlers keep emitting `gin.H{"error":"string"}` and the frontend parser handles both shapes. Future phases can migrate more endpoints incrementally.

**Tech Stack:** Go 1.25 / Gin (envelope already present); react-i18next v17 (namespace registration).

---

## Scope guardrails

- **Only ~10 endpoints migrate.** The survey identified the high-visibility set (login, settings PUT, library rename/move/delete, library new folder, upload finalize, digest reset/trigger, search). Everything else keeps the legacy flat shape and gets English fallback through the parser.
- **Static codes only.** No dynamic params / interpolation in Phase 3 — every migrated error is a static sentence. The one partial exception is `search` which currently leaks `{{query}}` length; keep the message static, don't interpolate length.
- **SSE / WebSocket / ACP streaming: excluded.** They use their own framing.
- **No panic-recovery middleware change.** Separate concern; not in scope.
- **No new error codes needed in `response.go`.** We add a second, semantic-code helper — see Task 1.
- **Both en and zh-Hans get complete coverage for migrated codes.** Same discipline as Phase 2.

// RespondCoded sends an error response with an explicit semantic code.
// Use this for user-facing errors where the frontend will translate the code
// to a localized message via the `errors` i18n namespace.

func RespondCoded(c *gin.Context, status int, code ErrorCode, message string) {
	respondError(c, status, code, message, nil)
}

**Step 1: Write the parser + formatter**

`frontend/app/lib/errors.ts`:

```ts
import i18n from '~/lib/i18n/config';

/** Shape emitted by RespondCoded and the legacy flat shape. */
export interface ApiError {
  code?: string;
  message: string;
}

/**
 * Read an error response body and normalize both the new envelope
 * ({ error: { code, message } }) and the legacy shape
 * ({ error: "string" }) into a single ApiError.
 */
export async function parseApiError(response: Response): Promise<ApiError> {
  let body: unknown = null;
  try {
    body = await response.clone().json();
  } catch {
    // Non-JSON body (HTML, empty, etc.) — fall through
  }

  // New envelope: { error: { code, message, details? } }
  if (
    body &&
    typeof body === 'object' &&
    'error' in body &&
    body.error &&
    typeof body.error === 'object' &&
    'message' in body.error
  ) {
    const err = body.error as { code?: string; message: string };
    return { code: err.code, message: err.message };
  }

  // Legacy flat: { error: "string" }
  if (body && typeof body === 'object' && 'error' in body && typeof body.error === 'string') {
    return { message: body.error };
  }

  // Nothing usable — fall back to status text
  return { message: response.statusText || 'Request failed' };
}

/**
 * Translate an ApiError via the `errors` namespace, falling back to the
 * backend's English `message` when the code is missing or untranslated.
 * Accepts a `t` function from useTranslation (preferred inside components)
 * or falls back to the i18n singleton for use outside render paths.
 */
export function formatApiError(
  err: ApiError,
  t?: (key: string, opts?: { defaultValue: string }) => string
): string {
  const translator = t ?? ((key: string, opts?: { defaultValue: string }) => i18n.t(key, opts));
  if (err.code) {
    return translator(`errors:${err.code}`, { defaultValue: err.message });
  }
  return err.message;
}
```

**Step 2: Write the hook**

`frontend/app/hooks/use-error-message.ts`:

```ts
import { useTranslation } from 'react-i18next';
import { formatApiError, type ApiError } from '~/lib/errors';

/**
 * Format an ApiError into a localized string. Pass the result of
 * `parseApiError(response)` or any `{ code?, message }` pair.
 *
 * Example:
 *   const tErr = useErrorMessage();
 *   const apiErr = await parseApiError(response);
 *   toast.error(tErr(apiErr));
 */
export function useErrorMessage() {
  const { t } = useTranslation('errors');
  return (err: ApiError) => formatApiError(err, t);
}
```

## Risks & notes

- **Legacy shape coexistence**: `parseApiError` handles both shapes. As long as unmigrated handlers still emit `gin.H{"error":...}`, the frontend silently falls back to the English message (acceptable — Phase 2's zh-Hans UI will have these English errors as "holes" until future migration).
- **`AUTH_INVALID_TOKEN` vs `AUTH_INVALID_SESSION`**: identical user-facing message today. Keeping separate codes because the backend already uses them to hint at different triggering conditions; frontend UX may later diverge (e.g., different retry behavior).
- **No breaking change for current users**: all UI still works; errors just become localized for migrated endpoints.
- **Migration doesn't require DB changes**: settings are untouched.
- **The `AUTH_*` messages are slightly reworded in i18n catalogs** ("Your session is no longer valid. Please sign in again.") to be more user-friendly than the backend's "Unauthorized". The backend still sends "Unauthorized" as `message`, but the code resolves to the friendlier string via `errors:AUTH_INVALID_TOKEN`. This is by design: backend messages are debugging fallbacks, not UX copy.
