# i18n Design

**Date:** 2026-04-21
**Status:** Approved
**Scope:** Web frontend + Apple native shell. Docs site deferred.

## Goals

Add internationalization to MyLifeDB so the UI can render in English or Simplified Chinese, with architecture extensible to more locales without refactor.

## Non-goals (v1)

- Starlight docs site i18n (deferred)
- Languages beyond `en` and `zh-Hans` (architecture supports; content not authored)
- RTL support (not needed for en/zh-Hans)
- SSR for SEO (SPA app, not relevant)
- Translation management system — JSON files in git sufficient for 2 locales + solo user
- AI-assisted translation workflow — translations hand-authored for v1

## User preference model

Add a new field distinct from the existing `preferences.languages[]` (which is for content tagging):

```ts
preferences.language: 'en' | 'zh-Hans'  // UI language, single value
```

Stored in SQLite under key `preferences_language`. This single value drives the web UI, the Apple native shell, and the WebView inside the Apple app. First-load defaults: `preferences.language` if set → `Accept-Language` header / system locale → `en`.

## Architecture

```
┌─────────────────────────────────────────────────────────────┐
│  User preference:  preferences.language ∈ {en, zh-Hans}     │
│  Stored in SQLite (settings table, key: preferences_language)│
└──────────────────────┬──────────────────────────────────────┘
                       │
        ┌──────────────┼──────────────────┐
        │              │                  │
        ▼              ▼                  ▼
   ┌─────────┐   ┌──────────┐      ┌─────────────┐
   │ Web SPA │   │ Apple    │      │ Backend Go  │
   │ react-  │   │ native   │      │ error codes │
   │ i18next │   │.xcstrings│      │ + msg fallback│
   └─────────┘   └──────────┘      └─────────────┘
```

Single source of truth: `preferences.language`. Web reads it on load; Apple reads it at launch; backend error codes get translated client-side based on it.

## Web frontend (React)

### Library choice: `react-i18next`

Rationale:
- Most mature, most popular React i18n library (~4M weekly npm downloads, 8.5k+ stars)
- Actively maintained by the i18next org
- Broad ecosystem (http-backend, browser detection, ICU plugin)
- Best Stack Overflow coverage for edge cases

Alternatives considered:
- `react-intl` / FormatJS — ICU MessageFormat is a standard, but DX heavier
- `@lingui/react` — smaller runtime but less mature, smaller ecosystem
- Hand-rolled — rejected; CLDR plural rules, formatters, extraction tooling all become maintenance burden by language #3

### Bundling strategy

**Bundle English fallback inline + locale metadata; lazy-load everything else.**

Rationale: the app already ships an SPA with React + Router + Tailwind. Bundling N locales adds ~5–15KB per locale to every cold load for zero benefit — most users only use one language. Instead:

1. English catalog inlined in main bundle (~5KB) — used as fallback when a key is missing and as instant-render safety net
2. All other locales lazy-loaded from `/locales/{lang}/*.json`
3. Active locale preloaded via `<link rel="preload" as="fetch">` injected in HTML based on `preferences.language` cookie/SQLite read. Fetch starts in parallel with JS parsing; no user-visible flash
4. Tiny locale-metadata file (`{ en: "English", zh-Hans: "简体中文", ... }`, ~few hundred bytes) bundled so the language picker renders native names without network

Native Apple side takes the opposite approach: bundle all locales (App Store download paid once at install).

### File layout

```

├── app/

│   │   ├── config.ts              # i18next init
│   │   ├── languages.ts           # (existing) BCP 47 helpers
│   │   ├── metadata.ts            # native-name metadata (tiny inline)

│   └── ...
public/
└── locales/
    ├── en/
    │   ├── common.json            # nav, buttons, generic
    │   ├── settings.json          # /me page
    │   ├── data.json              # file browser, collectors
    │   ├── agent.json
```

Namespaces split by route/feature so each page lazy-loads only what it needs.

### Usage pattern

```tsx
const { t } = useTranslation('settings');
<button>{t('save')}</button>
<p>{t('items', { count })}</p>   // pluralization via i18next CLDR rules
```

### First-load flow

1. Go backend reads `preferences_language` from SQLite on `GET /` (and SPA routes), injects into HTML template:
   - `<html lang="zh-Hans">`
   - `<link rel="preload" as="fetch" href="/locales/zh-Hans/common.json">`
2. React boots; i18next reads lang from `<html>` element and starts loading preloaded catalog
3. First render uses English fallback → swaps to active locale when loaded (typically before first paint)
4. No flash of keys

### Language switcher

Added to `/me` General tab, adjacent to existing `LanguageSelector` (content languages). On change:
- Writes to `preferences.language` via `PATCH /api/settings`
- Calls `i18n.changeLanguage(code)` — live UI update, no reload
- Persists to SQLite for subsequent sessions

## Backend (Go)

### 1. Locale-aware index.html

- Backend reads `preferences_language` from SQLite on `GET /` and any SPA route
- Injects into HTML template: `<html lang="{lang}">` + preload hint for `/locales/{lang}/common.json`
- If unset, defaults to parsing `Accept-Language` header → fallback `en`

### 2. Error codes

New error envelope for user-facing handlers:

```go
type APIError struct {
    Code    string `json:"code"`    // e.g. "people.fetch_failed"
    Message string `json:"message"` // English fallback
}
```

**Code conventions:**
- Dot-separated: `{domain}.{action}_{result}` — e.g. `settings.save_failed`, `files.upload_too_large`, `auth.invalid_credentials`
- Defined as constants in `backend/api/errors.go`, grouped per domain
- v1 migrates only user-facing high-visibility handlers (settings, auth, file upload, validation). Internal/rare errors stay plain-string; frontend falls back to the `message` field (English). Long-tail translation can be added later without another architectural refactor.

**Frontend consumption:**
```ts
// locales/{lang}/errors.json
{
  "people.fetch_failed": "Failed to load people",
  "settings.save_failed": "Could not save settings"
}

// hook
const msg = useErrorMessage(err); // maps code → translated, falls back to err.message
```

## Apple native (SwiftUI)

### Localization format: `.xcstrings`

Modern Apple String Catalog (Xcode 15+). One file `Localizable.xcstrings` with all keys + per-locale translations + pluralization via Xcode's native UI.

### Locale resolution at launch

Hybrid — system default with user override:

```swift
// Pseudocode
let systemLocale = Locale.current.language.languageCode
let userPref = await api.fetchPreferences().language  // nil if unset
let active = userPref ?? mapSystemToSupported(systemLocale) ?? "en"
Bundle.setLanguage(active)  // via SwiftUI .environment(\.locale) override
```

Flow:
1. App launches → set locale from `UserDefaults` cache (instant, no flicker)
2. In parallel: fetch `/api/settings` → update `preferences.language` cache if changed
3. SwiftUI re-renders with new locale

### In-app language switcher

Row in `Me > General` — writes to `preferences.language` via API, updates locale live, persists to UserDefaults.

### WebView inside the app

Loads the React SPA with the same `preferences.language`. Web-side flow handles locale from there. Native shell and WebView stay in sync because both read the same preference.

### v1 strings to translate

~30–50 keys total:
- Tab names: Data / Agent / Explore / Me
- Me view: General / Data Collect / Server / Stats / About / Sign Out
- Login: MyLifeDB tagline, Sign In, Done

## Formatting (dates, numbers, plurals)

Strategy: use `Intl.*` APIs directly, wrapped in tiny helpers that read the active locale from i18next.

Helpers in `frontend/app/lib/i18n/format.ts`:
```ts
formatDate(d, style?)       // "Apr 21, 2026" / "2026年4月21日"
formatRelative(d)           // "2 hours ago" / "2小时前"
formatNumber(n, opts?)
formatFileSize(bytes)
```

All read `i18n.language` internally. No component passes locale.

### Existing code migration

- `formatTimestamp` in `lib/utils/format-timestamp.ts` — replace hardcoded "Yesterday" with locale-aware relative formatter (`Intl.RelativeTimeFormat`)
- `date-fns formatDistanceToNow` in `pending-file-card.tsx` — switch to `Intl.RelativeTimeFormat`, no extra dependency, works for all future locales automatically
- Bare `toLocaleString()` / `toLocaleDateString()` calls — pass active locale explicitly

### Pluralization

i18next's built-in CLDR plural rules:
- `t('files', { count })` → English: `"1 file"` / `"3 files"`; Chinese: `"3 个文件"` (single form)

### Apple side

SwiftUI's `Text(Date(), style: .relative)` + `String(localized: "\(count) files")` — Apple's native formatters handle all CLDR rules. Zero custom code.

## Rollout phases

Each phase is independently shippable. App works in English throughout phases 1, 3, 4, 5; zh-Hans users get progressively more translated surface area.

### Phase 1 — Web foundation (no visible UI change)
- Install `react-i18next` + `i18next-http-backend`
- Create `lib/i18n/config.ts`, namespace structure
- Extract English strings from high-traffic files into `public/locales/en/*.json` (keys only, no zh-Hans yet)
- Wire `useTranslation` throughout; verify app still renders identically in English
- Add Go handler to inject `<html lang>` + preload hint

### Phase 2 — Add Simplified Chinese
- Author `public/locales/zh-Hans/*.json`
- Add language switcher in `/me` General tab
- Test switching, persistence, fresh-load with `preferences.language=zh-Hans`

### Phase 3 — Backend error codes
- Define `APIError` envelope, migrate user-facing handlers
- Add `errors.json` catalog, `useErrorMessage` hook
- Replace frontend's direct `err.message` displays

### Phase 4 — Formatters
- Add `lib/i18n/format.ts` helpers
- Migrate `formatTimestamp`, `date-fns`, bare `toLocale*` calls

### Phase 5 — Apple native
- Create `Localizable.xcstrings` in Xcode
- Extract SwiftUI strings (Tab names, Me view, Login)
- Add locale resolution at launch + in-app switcher
- Translate to zh-Hans

## Testing

- **Unit:** helpers for `formatDate`, `formatRelative`, error-code mapping
- **Component:** snapshot tests of key pages in both locales
- **Integration:** Playwright — set `preferences.language=zh-Hans`, reload, verify nav shows 数据/代理/探索/我
- **Manual:** visual QA of both locales across main flows
- **Apple:** Xcode preview variants for both locales + on-device smoke test
