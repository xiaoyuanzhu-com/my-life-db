# i18n Phase 2 — Simplified Chinese + Language Switcher

**Goal:** Ship a user-selectable UI language. Add `preferences.language` (singular) end-to-end (TS type, Go model, DB row, sanitize, API), author zh-Hans catalogs covering every key shipped in Phase 1, and add a language switcher in `/me` → General that persists to the backend and swaps catalogs live without a full reload.

**Architecture:** Locale resolution lives server-side (Phase 1 already reads `preferences_language` and injects `<html lang>` + preload hints). Phase 2 adds the user-facing controls: a singular `preferences.language` field distinct from the existing `preferences.languages[]` (which describes content languages, unchanged). The switcher writes through the settings API → DB → next request's `<html lang>` matches, and client-side calls `i18n.changeLanguage(next)` so the current session re-renders without refresh.

**Tech Stack:** TypeScript / React / react-i18next v17 (already configured); Go 1.25 / Gin / SQLite; `golang.org/x/text/language` (already vendored).

---

## Scope guardrails

- **zh-Hans is the only new translation** — en remains source of truth. No other languages.
- **Translate only what's already keyed** — Phase 1 inlined ~87 keys into `public/locales/en/*.json` + `app/lib/i18n/fallback/en.json`. Translate those exact keys. Do not extract new strings; that's Phase 3+ work.
- **`preferences.language` is new; `preferences.languages[]` is untouched** — different purposes (UI chrome vs. content language tags). Both coexist.
- **Live language switch** works via `i18n.changeLanguage()`. No page refresh required for the user. But on next hard load, `<html lang>` will match because the backend reads the saved setting.
- **No translation memory / no automated translation pipeline.** Hand-authored zh-Hans by the implementer for the ~87 keys. Locale-idiomatic, not literal. Keep placeholders (`{{name}}`, `{{count}}`) and pluralization suffixes (`_one`/`_other`) intact.

**Step 2: Write the switcher component**

`frontend/app/components/settings/ui-language-selector.tsx`:

```tsx
import { useTranslation } from "react-i18next";
import i18n from "~/lib/i18n/config";
import {
  SUPPORTED_UI_LOCALES,
  LOCALE_NATIVE_NAMES,
  type UiLocale,
} from "~/lib/i18n/metadata";

interface UiLanguageSelectorProps {
  value: string | undefined;             // preferences.language (may be undefined/empty)
  onChange: (value: string | undefined) => void;
}

export function UiLanguageSelector({ value, onChange }: UiLanguageSelectorProps) {
  const { t } = useTranslation("settings");

  const handleChange = (next: string) => {
    const normalized = next === "" ? undefined : (next as UiLocale);
    onChange(normalized);
    // Swap catalogs immediately so the user sees the change without a page reload.
    // If normalized is undefined, fall back to the locale injected at boot.
    const target = normalized ?? document.documentElement.lang ?? "en";
    void i18n.changeLanguage(target);
  };

  return (
    <select
      className="w-full px-3 py-2 rounded-md border bg-background"
      value={value ?? ""}
      onChange={(e) => handleChange(e.target.value)}
    >
      <option value="">{t("general.uiLanguage.systemDefault", "System default")}</option>
      {SUPPORTED_UI_LOCALES.map((code) => (
        <option key={code} value={code}>
          {LOCALE_NATIVE_NAMES[code]}
        </option>
      ))}
    </select>
  );
}
```

Rationale:
- Selector uses the i18n singleton for `changeLanguage` because this runs outside a render path when the user picks an option. (Same pattern Phase 1 used in `settings-context.tsx` for async toasts.)
- "System default" maps to `undefined` in settings state, which means the DB row is omitted and the backend falls back to `Accept-Language` → English.
- Live swap via `i18n.changeLanguage` — no page reload. `<html lang>` only updates on next hard load, which is fine: it's used for SEO/accessibility and boot-time resolution, not live UI.

## Risks & notes

- **Key drift between locales**: If zh-Hans is missing a key, english fallback kicks in — which is the correct behavior. The verification step guards against accidental extra keys in zh-Hans (clutter).
- **CLDR pluralization for Chinese**: Chinese has only `other` in CLDR, but react-i18next accepts `_one` as a synonym when keys exist. Keeping both prevents log noise.
- **`changeLanguage` race on save**: The selector calls `changeLanguage` immediately; the save happens later when the user clicks "Save Settings". If they cancel/navigate away without saving, the session has swapped locale but the DB wasn't updated — next reload reverts. That's acceptable (matches how most "live preview" settings behave).
- **No migration needed**: `settings` is a flat KV table; new key `preferences_language` just appears when first written.
